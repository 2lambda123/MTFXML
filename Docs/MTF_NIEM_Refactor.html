NIEM RE-FACTOR DECISION PAPER
11 Aug 2017
Subj:     NIEM RE-FACTOR DECISION PAPER
1.  Background.  Beginning in 2015 the USMTF Configuration Control Board has been refactoring  XML MTF to achieve NIEM Conformance. 
2.  NIEM Conformance.  The decision to refactor XML MTF to be conformant to NIEM Reference Schema Naming and Design rules was approved by the CCB in 2015.
3.  Prototype Schema.  USMC has generated NIEM Conformant Reference Schema to demonstrate complete representation of the MIL STD.  The attachment provides counts of all required changes and lists each changed Element
4.  Impact on Instance Doucments.  Some XML Element Names must be changed changed in order to prevent duplication at the global level, to conform to NIEM Camel Case requirements, or to employ required representation terms.  This results in changes in XML instance documents, which impact software that does not directly reference the XML Schema for field names and will require updating.  The attached spreadsheet documents the number of changes, the reason for the changes, the number of changes per message, and a complete listing of all elements that require changes.
5.  Rationale for Changes
      a.  CamelCase.  The design of existing XML MTF Schema required that some Elements be identified uniquely by appending an underscore with a number.  This naming is not NIEM conformant for Camel Case naming and is not required when using NIEM naming and design rules. 
      b.  General Text.  All GenText Elements must be unique at the global level.  This is done by pre-pending a camel case concatenation of the required Text Indicator statement.  Since most GenText Elements have appended numbers, they violate Camel Case naming requirements and must be changed anyway.
      c.  Heading Information.  All HeadingInformation Elements must be unique at the global level.  This is done by pre-pending a camel case concatenation of the required Text Indicator statement. Since most HeadingInformation Elements have appended numbers, they violate Camel Case naming requirements and must be changed anyway.
      d.  Field Groups.  The GroupOfFields elements must be unique at the global level.  For single item groups, the GroupOfFields element is eliminated and cardinality applied to the single element.  For multiple item groups a FieldGroup element is created by appending "FieldGroup" to the name of the containing Set.
      e.  Substitution Groups.  The use of substitution groups requires that each member has a unique name, and cannot be a member of more than one substitution group.  These names are only changed when required by pre-appending the name of the parent Entity.
6.  Impact Analysis.  Because Naming Changes required due to Camel Case rules can either remain, or be accommodated by removing underscore numbers and appending consistent text, the impact may be less than those names that must be changed due to conflicts, field group modification, and substitution group membership.  Services must evaluate impact on current implementations based on specific messages affected, the types of changes, and the potential difficulty in accomodating those changes.
7.  Courses of Action
      a.  1. Submit Fields, Composites, Sets, Segments and Messages for inclusion in MIL OPS Domain using NIEM Reference Schema conformant names.  Field NIEM conformant MTF as a new version for all new implementations.  Provide transformations to convert instances to and from older versions.
          (1)  PRO:.  Leverages NIEM methodology for the implementation of MTF using required external functions.  These include security tagging, alignment with semantic reference models, and interoperability with other message formats.
          (2)  CON:.  Requires XML instance names changes that will inhibit implementation by existing software.
      b.  2.  Continue to evaluate options with regard to changing NIEM naming and design rule requirements, reducing impact on current message XML instances, and accomodating backward compatibility.
          (1)  PRO:.  Provides time for services and vendors to evaluate impact on existing software and propose further XML Schema designs.
          (2)  CON:.  Ignores three years of the MTF CCB process, delays efforts to achieve NATO alignment, and may not result NIEM conformant XML Schema that can be used to fully leverage the NIEM ecosystem.
      c.  3. Submit only Fields and Composites for inclusion in MIL OPS Domain using NIEM Reference Schema conformant names.
          (1)  PRO:.  Provides MTF fields to the MIL OPS domain for external use by NEIM conformant XML Schema without impacting MTF implementation.
          (2)  CON:.  Does not leverage NIEM methodology to implement MTF or enable external functions.
8.  Recommended Course of Action: COA 1.  Existing software that cannot adjust Element names by referencing XML Schema represents design decisions that should not be encouraged or propagated.  The advantages of the NIEM methodology far outweigh the impact on legacy software.
